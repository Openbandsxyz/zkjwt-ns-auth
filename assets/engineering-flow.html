How NS Auth works

A step-by-step walkthrough of every request, redirect, and database operation in the NS OAuth system.

Overview The Flow Security Tokens Scopes & Data Database Errors API
01

The big picture

Four systems talk to each other. The user never leaves their browser.

The user clicks "Sign in with Network School" → authenticates via Discord → approves scopes → gets tokens back. That's it.

02

The flow, step by step

We'll walk through the entire authorization code + PKCE flow in 6 phases. Each phase has a diagram showing exactly what happens.

Phase A — User clicks "Sign In"

1

CLIENT Generate PKCE pair

Create a random code_verifier (43-128 chars). Hash it: code_challenge = base64url(SHA256(verifier)). Also generate a random state for CSRF.

2

CLIENT Redirect to NS Auth

Browser navigates to /oauth/authorize with query params: response_type=code, client_id, redirect_uri, scope, state, code_challenge, code_challenge_method=S256.

3

BACKEND Validate everything

Check: client_id exists? redirect_uri matches registered? scopes are valid? code_challenge_method is S256? All good → check for ns_session cookie.

4

BACKEND No session → redirect to login

No cookie found. 302 redirect to the frontend's /login page, forwarding all OAuth params so they survive the login round-trip.

Phase B — Discord login

5

BACKEND Create CSRF state

Generate a random 43-char state token. Store it in oauth_states table with a 10-minute TTL and the next URL (where to go after login). This prevents CSRF attacks on the callback.

6

BACKEND Redirect to Discord

302 to discord.com/oauth2/authorize requesting scopes: identify email guilds.members.read. The state token goes along for the ride.

7

DISCORD User logs in

User authenticates with Discord (username + password, or if already logged in, just click "Authorize"). Discord redirects back to our /auth/discord/callback with a code and the state.

8

BACKEND Validate callback

Check the state matches what's in our DB (one-time use — delete it). Then exchange the Discord code for a Discord access token by calling POST discord.com/api/v10/oauth2/token.

9

BACKEND Fetch Discord profile

Use the Discord token to call GET /users/@me → get the user's Discord ID, username, email, avatar, and public flags.

Phase C — Membership gate

10

BACKEND Verify NS membership

Using the bot token (not the user's token), call GET /guilds/{guild_id}/members/{discord_id}. If Discord returns 404 → the user is not in the NS server → 403 not_ns_member. No exceptions.

11

BACKEND Create or update user

Look up users by discord_id. First time? Insert a new row. Returning user? Update their email if it changed. This is JIT (just-in-time) provisioning — no user registration flow needed.

Phase D — Session handoff

We need to get a session JWT to the frontend. But we can't put the JWT in the URL — it would leak in browser history, server logs, and Referer headers. So we use a session code relay:

12

BACKEND Generate session code

Create a 43-char random code, store it in session_codes table with the user_id and a 60-second TTL. This code is single-use.

13

BACKEND Redirect to frontend

302 to {frontend}/auth/session?code={session_code}&next=/consent?.... The short-lived code is safe to put in the URL — even if leaked, it expires in 60 seconds and can only be used once.

14

FRONTEND Exchange code for JWT

POST /auth/session/exchange with {"code": "..."}. Backend validates: exists? not expired? not already used? → marks it used → returns an HS256 session JWT (24h TTL). Set as ns_session httponly cookie + localStorage.

Phase E — User consent

15

FRONTEND Show consent screen

Fetch app info from GET /oauth/authorize/info and user from GET /auth/me. Display the app name, its icon, and which scopes it's requesting. User clicks "Allow" or "Deny".

16

BACKEND Generate authorization code

On "Allow": generate a 128-char auth code (secrets.token_urlsafe(64)). Store in authorization_codes with: user_id, app_id, scope, code_challenge, redirect_uri. Expires in 10 minutes.

17

BACKEND Return redirect URL as JSON

Return {"redirect_to": "client_callback?code=...&state=..."}. Why JSON and not 302? Because the frontend made a fetch() with cookies. A cross-origin 302 creates an opaque redirect — JavaScript can't read the Location header. So we return it in the body.

Phase F — Token exchange

18

CLIENT Exchange code for tokens

POST /oauth/token with grant_type=authorization_code, the auth code, the original code_verifier, client_id, and client_secret.

19

BACKEND Verify everything and issue tokens

Five checks: (1) client_id + secret match (bcrypt), (2) auth code exists and not expired, (3) code not already used, (4) redirect_uri matches, (5) PKCE: SHA256(code_verifier) == stored code_challenge. Then mark code used and issue:

Access Token

RS256 JWT, 1h TTL. Contains sub, iss, aud, scope, jti. SHA256 hash stored in DB.

Refresh Token

Opaque 64-char string, 30-day TTL. SHA256 hash stored with family_id for rotation.

ID Token

RS256 JWT (only if openid scope). Contains sub, iss, aud, nonce.

20

CLIENT Fetch user data

GET /oauth/userinfo with Authorization: Bearer {access_token}. Backend introspects the token, fetches live data from Discord API (cached 5 min), and returns claims based on granted scopes.

Flow complete.

The client app now has tokens and user data. The user is signed in with Network School.

03

Security mechanisms

Eight layers of protection. Click any to see how it works.

PKCE S256 Prevents authorization code interception

The client sends a hashed version in step 2 and the original in step 18. An attacker who intercepts the authorization code can't use it without the original verifier. The plain method is disabled — only S256.

CSRF STATE DB-stored, 10-min TTL, single-use

Before redirecting to Discord, we generate a random state and store it in PostgreSQL with a 10-minute expiry. On callback, we look it up — if missing, expired, or wrong → reject. The row is deleted on first use, preventing replay.

state = secrets.token_urlsafe(32)  // 43 chars
INSERT INTO oauth_states (state, next_url, expires_at)
VALUES (state, next_url, now() + interval '10 min')
SESSION RELAY 60-second single-use code, not JWT in URL

JWTs in URLs leak through browser history, Referer headers, and server logs. Instead we redirect with a short-lived code that the frontend exchanges server-side for the actual JWT. Even if the code leaks, it's useless after 60 seconds or first use.

REPLAY DETECTION Auth code reuse → revoke everything

Authorization codes have a used flag. If someone tries to exchange an already-used code, we assume the code was stolen and revoke all tokens that were issued from it. This is per RFC 6819.

if code.used == True:
    // REPLAY ATTACK DETECTED
    revoke all access_tokens where code_id = code.id
    revoke all refresh_tokens where code_id = code.id
    return 400 "invalid_grant"
REFRESH ROTATION Family tracking + chain revocation

Every refresh token use produces a new token. All tokens in a chain share a family_id. If a previously-revoked refresh token is presented (stolen token being replayed), the entire family is revoked — attacker and legitimate user both lose access, forcing re-authentication.

RT1 (family_id=abc) → used → RT2 (family_id=abc) → used → RT3 (family_id=abc)

Attacker replays RT1:
  RT1 is already revoked → THEFT DETECTED
  → Revoke RT2, RT3, and all access tokens with family_id=abc
RS256 4096-BIT Asymmetric token signing

Access tokens and ID tokens are signed with a 4096-bit RSA key. Anyone can verify tokens using the public key at /.well-known/jwks.json without calling our server. Key ID (kid) is derived from SHA256 of the DER-encoded public key (first 8 hex chars).

Dev: Keys auto-generated to backend/keys/ on first startup.
Prod: Base64-encoded PEM in env vars (OAUTH_RSA_PRIVATE_KEY, OAUTH_RSA_PUBLIC_KEY).

TOKEN HASHING SHA256 for tokens, bcrypt for secrets
SecretAlgorithmWhy
client_secretbcrypt (12 rounds)Low entropy, user-facing. Needs salt + cost factor.
access_tokenSHA256High entropy (JWT). Fast lookup sufficient.
refresh_tokenSHA256High entropy (64 chars). Needs fast comparison for rotation.
MEMBERSHIP GATE Non-NS-members cannot proceed

After Discord login, backend calls the Discord API with the bot token to check if the user is in the NS guild. This isn't optional — non-members get a hard 403 and cannot create a session. The bot needs the SERVER MEMBERS INTENT enabled.

Scopes & claims

Scopes gate what data the client can access. Most data comes live from Discord — nothing is cached in our DB.

ScopeClaims returnedSource
openidsubDB User UUID
emailemail, email_verifiedDB Stored at first login
profilename, picture, discord_username, banner_url, accent_color, public_badgesDiscord Live 5-min cache
rolesroles (array of role names)Discord Live 5-min cache
date_joineddate_joined, discord_joined_at, boosting_sinceDB + Discord
offline_access—Enables refresh tokens. No claims.
How does the Discord data cache work?

On every /oauth/userinfo call, the backend fetches the user's Discord profile using the bot token (not the user's OAuth token). Results are cached in an in-memory LRU cache:

Cache typeIn-memory LRU (cachetools.TTLCache)
Max entries1,000
TTL5 minutes
KeyDiscord user ID

This means role changes, avatar updates, etc. are reflected within 5 minutes. The cache is per-process (not shared across instances).

OAuth

GET /oauth/authorize — Start authorization flow

Params: response_type=code, client_id, redirect_uri, scope, state, code_challenge, code_challenge_method=S256, nonce (optional).

→ 302 to frontend /login (no session) or /consent (has session).

POST /oauth/token — Exchange code / refresh / client_credentials 
// Authorization code grant
POST /oauth/token
grant_type=authorization_code&code=...&redirect_uri=...&code_verifier=...&client_id=...&client_secret=...

// Response
{ "access_token": "eyJ...", "token_type": "Bearer", "expires_in": 3600,
  "refresh_token": "...", "id_token": "eyJ...", "scope": "openid profile" }

// Refresh grant
grant_type=refresh_token&refresh_token=...&client_id=...&client_secret=...

// Client credentials (no user context)
grant_type=client_credentials&client_id=...&client_secret=...&scope=openid

Authentication

GET /auth/discord — Start Discord login

Param: next (post-login redirect URL). Generates CSRF state → stores in DB → 302 to Discord OAuth2.

GET /auth/discord/callback — Discord OAuth2 callback

Discord redirects here with code + state. Validates state, exchanges code, fetches profile, checks guild membership, provisions user, creates session code, 302 to frontend.

Discovery

GET /.well-known/openid-configuration

Standard OIDC discovery document. Lists all endpoints, supported grant types, scopes, and signing algorithms.

GET /.well-known/jwks.json

RSA public key in JWK format. kid = first 8 hex chars of SHA256(DER public key).
